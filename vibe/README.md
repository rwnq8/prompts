# The V-Series Protocol: PMBOK for Vibe Coding

**Structured Project Management for LLM-Assisted Software Development**

The **V-Series Protocol** is a set of three strict System Prompts designed to transform the chaotic process of "Vibe Coding" (rapid, LLM-driven development) into a disciplined engineering workflow.

By applying **PMBOK (Project Management Body of Knowledge)** phases—Planning, Execution, and Monitoring—to AI generation, this protocol ensures that applications are robust, documented, and secure, rather than just "barely working."

---

## 📂 The Three Agents

The system operates via three distinct persona prompts. You do not use them all at once; you chain them sequentially.

### 1. `V1_Architect.md` (The Planner)
*   **Role:** Chief Solutions Architect.
*   **Phase:** Planning & Design.
*   **Input:** Your raw idea, "vibes," or complex requirements.
*   **Output:** A strict **Engineering Blueprint**.
    *   Defines Tech Stack.
    *   Maps File Structure.
    *   Specifies Data Flow.
    *   **Crucial:** Does NOT write code.

### 2. `V2_Builder.md` (The Executor)
*   **Role:** Senior Full-Stack Developer.
*   **Phase:** Implementation.
*   **Input:** The Blueprint from V1.
*   **Output:** **Production-Ready Code**.
    *   Writes all files.
    *   **Crucial:** Automatically generates "Living Documentation" (`REQUIREMENTS.txt`, `DEVLOG.md`, `ROADMAP.md`).
    *   **Crucial:** Implements "Local RAG" logic (reading from a `/docs` folder) if required.

### 3. `V3_Auditor.md` (The Controller)
*   **Role:** QA & Security Engineer.
*   **Phase:** Monitoring & Control.
*   **Input:** The Code from V2.
*   **Output:** **Audit Report & Test Suite**.
    *   Checks logic against Requirements.
    *   Scans for Security vulnerabilities.
    *   Updates the `DEVLOG` and `ROADMAP`.
    *   Generates `tests/`.

## 🚀 How to Use

### Step 1: Planning
1.  Open your LLM workspace (Cursor, Replit Agent, ChatGPT, Claude).
2.  Paste the content of **`V1_Architect.md`** into the System Prompt (or first message).
3.  Type your request: *"Build me a Python-based patent generator that uses these specific logic steps..."*
4.  **Result:** You get a detailed text file blueprint.

### Step 2: Execution
1.  *(Optional but Recommended)* Start a fresh context/chat to save tokens.
2.  Paste the content of **`V2_Builder.md`**.
3.  Paste the **Blueprint** generated by V1.
4.  **Result:** The LLM generates the full codebase, including configuration files and documentation.

### Step 3: Audit
1.  Paste the content of **`V3_Auditor.md`**.
2.  Paste the **Code** generated by V2.
3.  **Result:** The LLM reviews the code, provides fixes for bugs, updates the logs, and writes unit tests.

## 🌟 Key Features

### 1. Living Documentation Standard
Every project built with this protocol automatically includes and maintains:
*   **`REQUIREMENTS.txt`**: The immutable "Input Truth" (your original prompt).
*   **`DEVLOG.md`**: A running history of versions, changes, and audit results.
*   **`ROADMAP.md`**: Known issues and future features.

### 2. Local RAG (Knowledge Base)
The protocol natively supports "Local Retrieval-Augmented Generation."
*   If your app needs specific knowledge (e.g., *US Patent Law*, *D&D 5e Rules*, *Brand Style Guide*), V1 will architect a `/docs` folder.
*   V2 will write logic to read and use these files.
*   This allows you to build "Smart Apps" that reference static text files without external vector databases.

### 3. Stack Agnosticism
*   **V1** is trained to pick the "Path of Least Resistance."
*   If you need a CLI, it picks Python/Go.
*   If you need a Web App, it picks Next.js/React.
*   If you need Data Logic, it picks Streamlit/FastAPI.
